Day 5：在 vibe coding 時代打造規格驅動的生產級開發

可擴展工作流與團隊成長藍圖：從 vibe 原型到產品化落地

「Vibe Coding」不是「Vibe In Production」

前言

Google 軟體工程師的日常，這一年來幾乎是180度大轉變。2024年初以前，大家還花大量時間研究開發者 API 和文件，一行一行手動試寫程式，還得搞清楚 Python 判斷子字串用 substring in string、string.includes 還是 string.contains。

好不容易寫完，還要花大把功夫除錯、找出功能程式碼跟原始意圖之間的落差。現在，開發速度根本像開了火箭。團隊開始用 Coding Agents，例如 Antigravity1 或 Gemini CLI2，這些工具不只是給你文字建議，而是真的會操作工具、執行任務。

AI Coding Editor 可以在短時間內產出上千行有完整註解的程式碼，彷彿請了一堆永遠不會累、不會抱怨的實習生。不過，這裡有個陷阱：速度雖然爆表，但「速度幻覺」也很真實。

AI 寫程式的速度雖快，但 bug 產生的比例也可能跟著暴增。實作已經不再是主要瓶頸，反而可以善用 AI，在同樣時間內寫出比人類更全面的測試覆蓋。這種方式能大幅提升對程式碼的信心，後面會再詳細說明。

而且，AI 的潛力遠不只寫程式，還能幫忙系統設計、規格撰寫、規劃 roadmap、分析結果。現在的瓶頸，反而變成人類要怎麼整合和審查這些成果，不過其實相關工具也都已經到位。

「vibe coding」指的是用 AI 根據「vibe」或高層意圖，快速產生程式碼，而不是一行行手動硬寫。不過要強調，「vibe coding」不等於「vibe-in-production」。

我們雖然把 AI agent 的能力發揮到極致，但所有產出都經過完整規劃與控管，確保符合產品等級的穩定性，絕不會直接用沒驗證過的 AI 輸出。

在企業環境中，這種「vibe」其實就是所謂的 Agentic AI 開發。跟一般的生成式 AI 不一樣，Agentic AI 不只是聰明的自動補全，而是像混合型團隊成員一樣：用語言模型當大腦，搭配工具整合進工作流（就是雙手）。

它可以推理、寫規格、寫測試案例、用瀏覽器測 UI，甚至直接在 Git 上 commit 和 merge 程式碼。不過，當開發從個人實驗階段進入團隊大規模生產時，大家就會發現，午餐前寫完1000行程式，跟真正把軟體部署出去，完全是兩回事。

當 agent 出現「幻覺」時，也就是 AI 很自信地胡亂編造不存在的東西，它不只會產生一個 bug，還可能寫出上千行「vibe 一致」但功能壞掉的程式邏輯。

如果人類審查者被 AI 產生的大量 Pull Requests（PRs，讓同事在程式碼上線前先審查的請求）淹沒，寫程式的速度再快也沒用。這個流程根本沒有變快，只是把「待整理的東西」堆得更高而已。

幸好，還是有方法可以保護開發流程，但一定要一開始就用這些技巧，而不是做到一半才補救。這份白皮書會介紹一系列方法，從生產等級的程式開發，到程式碼安全，以及在 vibe coding 時代如何轉變團隊文化。

規格驅動開發（Spec-Driven Development, SDD）

在傳統的軟體工程領域，開發者通常被教導要「先寫程式碼」。有個模糊的想法，就直接打開編輯器開始寫，直到東西能跑為止。但進入 Agentic AI 時代後，日常工作型態已經大幅改變。

現在，大部分時間都花在撰寫高品質的規格，也就是那些能精確指示 AI 要做什麼的詳細技術說明。

在這種全新的工作流裡，開發者的角色更像是技術架構師，而不是傳統的程式設計師。重點在這裡：程式碼現在是可以隨時丟棄的。只要規格寫得夠扎實，整個程式碼庫都可以反覆重新產生。

甚至可以直接指示 agent 把整個專案從 Python 轉成 JavaScript，下午就搞定。因為程式碼是一次性產物，大家對它就不會有情感依賴。也因為沒有人花了十二小時只為了 debug 一個分號，所以當需求變動時，完全不用怕重來。

像 Antigravity 或 Gemini CLI 這類的 coding agents，運作方式是把大型語言模型（LLM）當作「大腦」來推理，再搭配工具當作「雙手」來執行任務。

如果你只給大腦一個「感覺 vibe」而不是明確的「藍圖 blueprint」，它就只能猜。而在企業級軟體裡，亂猜就是「Rogue Agent」事件發生的主因。

好的規格說明

那麼，什麼才算是符合生產等級的藍圖呢？一份好的規格說明就像是架構上的北極星，為整個專案指引方向。它能避免「脈絡破碎」的情況，這種情形就像玩傳話遊戲一樣，AI 只看到過時的檔案快照，漸漸搞不清楚重點。

其實，我們也可以用 AI 來協助完善規格，讓它擔任共同作者、規格編輯或審查者的角色。通常，規格會直接存放在程式碼庫裡，例如放在 specs/ 資料夾下的 Markdown 或 YAML 檔案。這份規格同時是人類與 AI 的「單一事實來源」。

應該用哪種格式？

2026 年，Ouyang 等人發表了一篇研究〈SkCC: Portable and Secure Skill Compilation for Cross-Framework LLM Agents〉3，發現 LLM agent 對於指令格式的敏感度極高。

如果只是用一般、沒特別優化的 Markdown 檔案，效能甚至可能掉到只剩六成。為了解決這個問題，研究團隊開發了 SkCC（Skill Compiler），這是一個超高速的工具，可以在 10 毫秒內，將單一來源的指令檔自動編譯成最適合模型的目標格式。

如果你們團隊用的是 Gemini，這篇論文強調，最理想的格式策略是「Markdown + 條件式 YAML」混合用法。Gemini 會用乾淨的 Markdown 標題來鎖定注意力，但只要遇到巢狀深度超過 3 層的結構化設定或資料 schema，切換成 YAML，效能就會明顯提升。

數據顯示，針對深層巢狀設定，YAML 的解析準確率高達 51.9%，遠勝過 JSON 的 43.1% 和 XML 的 33.8%。也就是說，把複雜巢狀規格用 YAML 呈現，敘述型指令維持在 Markdown，就能避開 JSON 帶來的「格式稅」，讓 Gemini 發揮最高準確度，同時也最省 token。

行為驅動

行為驅動開發（Behavior-Driven Development, BDD）規格，是把模糊不清的人類想法，轉換成 AI agent 能直接執行、毫不猜測的精確架構設計的終極利器。

BDD 的核心精神，就是在寫任何程式碼之前，用結構化、易懂的自然語言，從使用者角度明確描述系統應該怎麼運作。這是一種軟體開發方法論，讓需求溝通變得零誤會。

BDD 規格會用一套標準語法 Gherkin4。Gherkin 採用簡單、宣告式的模板：Scenario / Given / When / Then。這種格式會強迫 LLM 以狀態 > 動作 > 結果的邏輯思考，完全杜絕「vibe coding」，讓 agent 按照明確步驟執行。

一份適合生成新專案的規格，應該包含：

  • 完整技術設計：不要只寫「做一個登入頁」。要拆解成多個部分，清楚列出需求、資料庫 schema（資料結構），還有 API 規格（讓軟體各部分能溝通的「契約」）。
  • 視覺輔助：包含架構圖，還有明確列出要用的工具和函式庫及其版本。
  • 背景資訊：說明這個需求背後的「為什麼」。這會幫助 agent 預想後續步驟，讓它更能前瞻性思考。
  • 情境範例：描述什麼是「做得好」的狀況、哪些是錯誤情境，還要涵蓋邊界案例。

實用小技巧

我自己常用 Google Docs 撰寫技術設計，讓架構規劃可以被團隊成員反覆審查。這比以往更重要，因為你一定希望有同事先發現邏輯漏洞，而不是等到 AI 已經生成了幾千行錯誤程式碼才發現。

審查完之後，我會用「檔案 > 下載 > Markdown」功能，把文件存成 Markdown 檔，然後加到工作目錄的 specs/ 資料夾裡。

大型語言模型其實不會真正「理解」資料結構，它們只是在處理被切割成 token 的文字。你傳送的每一個字元，最後都會被拆成 token，而每個 token 都會消耗預算、時間和 context window 容量。

要寫出可上線的規格，就必須把 token 化視為一種硬性的物理限制。因為你送出的每個字、換行、縮排，最後都會直接影響你的開發預算和系統延遲。即使像 Google Antigravity 這種以 Gemini 為核心的 agent 平台，預覽期間有超大 context window 和內建配額，最終還是受限於底層模型的 token 運算規則。

每一個不必要的 YAML 深層縮排空格、每一個重複的 Given / When / Then 指令，都會在 agent 進行多輪推理、反覆建構與測試應用程式時，額外消耗處理資源和注意力容量。

所以，把 /specs 資料夾當成一組精簡、編譯過的指令集來經營，讓它在可讀的 Markdown 和高度聚焦、扁平的 YAML 區塊間取得平衡，就能大幅減少推理時的「格式稅」，讓 AI agent 走在精準又省成本的軌道上。

指令應該放在哪裡？

要實踐 SDD（規格驅動開發），首先要理解各種程式開發工具是怎麼讀取指令的。這些指令其實不會只寫在同一個地方。如果你直接把一份一百頁的系統設計文件丟進聊天視窗，不只會很快用光短期 context 預算，還會拉高延遲、讓 context 變得支離破碎。事實上，這些規格可以分散存放在不同位置。

1. 聊天介面（短暫存在、依 session 而異） IDE 裡的對話式輸入框（像是 Gemini 側邊欄或終端機介面），只會跟著當下開發者的 session 存在。

建議把聊天介面單純用來做高層次的協調和即時回饋循環。例如：「Review the design in specs/payment_retry.md and generate the failing unit tests defined in Scenario 3.」

spec 資料夾（任務導向、納入版本控制）這是一個直接放在程式庫裡的靜態資料夾，用來存放技術設計、BDD 情境、API 合約和結構化的 YAML schema。

agent 會動態索引這個目錄，自動建構和驗證程式，不需要手動把 prompt 塞進去。

./my-app/specs/my_spec.md

2. Agent Skills（可重用、聚焦功能或行為） Skills 是結構化的 Markdown 檔案，裡面包含專門針對觸發條件設計的工作流。Skills 可以放在程式庫的任何地方，但必須存放在指定的 .agent 目錄下，Antigravity workspace manager 才能辨識。

這些技能會教 agent 培養可重複的工程習慣，例如偵測到程式變動時，自動維護 CHANGELOG.md。

./my-app/.agent/skills/docs-maintenance/SKILL.md 這種 skills 資料夾裡，也可以放技能會用到的資料資產或腳本。

3. System Prompts（全域、聚焦身分識別）這裡是 AI 學習專屬工程 DNA 的地方。無論是 Gemini CLI 還是 Google Antigravity，都會用階層式方式掃描並串接 context 檔案，也就是說，客製化指令會從全域覆寫一路疊加到專案本地的設定。

  • Global Profile：這個檔案放在家目錄的設定資料夾（例如 ~/.gemini/GEMINI.md）。這裡會讓 AI 更貼近個人偏好，定義通用的人格、預設風格和核心原則，不管專案怎麼換都適用。
  • Shared Multi-Tool Config（AGENTS.md）：如果團隊同時用多種 AI 客戶端，為了避免指令分散，生態系支援 AGENTS.md。這個檔案是跨工具共用的基礎設定，而本地的 GEMINI.md 則保留最高優先權，專門處理 Google 相關設定。
./my-app/.agents/AGENTS.md)
  • Project Spec：這個檔案放在專案根目錄（例如 ./my-app/.gemini/GEMINI.md）。這就是專案的 DNA。CLI agent 會自動偵測並讀取這個檔案，並優先採用裡面的規則。

不同情境，不同 Prompt

把規格轉成程式碼，其實沒有唯一做法。這可以拆成幾種執行模式，每一種模式都需要不同的思維方式：

1. 專案產生（The Architect）：這個模式下，是從零開始做 scaffolding，建立專案的骨架。不能直接進入 YOLO Mode（亂碼一通），要明確在 prompt 裡要求 agent 先不要寫程式。

應該先提出資料夾結構和技術棧，等你確認後再動手。記得在 prompt 裡要求同時產生測試、文件，以及 logging（數位版的「黑盒子」，記錄 app 在做什麼）。

每個函式庫都要標明版本號。沒標的話，agent 可能會建議用舊版工具，因為它的「知識截止日」是在訓練資料收集完的那一天。

2. 功能開發（The Builder）這個模式是針對現有程式碼庫加新功能。要在 prompt 裡提醒 agent 跟現有風格一致，包含命名慣例、錯誤處理方式等。

如果一次編輯多個檔案，變更內容都要人工確認。最好能在編輯器裡直接看到「Diff」，也就是哪些行被新增、刪除，一目了然。

Bug 修復（The Forensic Specialist）出現問題時，就要進入 forensic mode，目標是找出根本原因，然後精準修補。要從「症狀式 prompt」（像是「按鈕沒反應」）切換成「證據式 prompt」（例如「用 gcloud logging read 'textPayload:ERROR' --limit 5 看到 403 錯誤」）。

可以用指令列工具（像是 gh 指令）來比對程式碼版本。接著說明流程：「Request 進入 Load Balancer → Auth 拿掉 header → Pod 失敗。」

實用小技巧

有時候 coding agent 會建議重新命名變數。這沒問題，但建議把變數命名獨立成另一個任務，不要和 bug 修復混在一起，這樣才不會造成後續混亂。

實用小技巧

有時候 coding agent 會建議用比較舊的工具、函式庫或模型版本。這是因為底層模型的訓練資料有時間截止點。例如它可能只建議 gemini-1.5-flash，因為模型裡沒有更晚的版本。

你可以用 coding editor 的 RAG 功能，把最新文件加進來，或是把文件下載成 markdown 放進 spec 資料夾。在 agent skills 或 profile prompt 裡也能補充這類資訊。

記得 agent 很容易回歸訓練資料，所以版本號一定要再三確認。

每次都要先要求有失敗的 unit test 或 curl 指令（就是發送資料到 URL 的方式）來重現 bug。這個測試案例要留在程式庫裡，確保 bug 不會回來。

嚴格限制 agent 只能修正根本原因，不要「順便」清理無關的程式，這樣才不會讓 code review 變得複雜。

想把 end-to-end（E2E）測試和除錯升級，可以用像 Antigravity 這種內建瀏覽器的工具，讓 agent 能自動跑 localhost、互動 UI、即時驗證畫面修正。

Antigravity 背後會自動啟動一個原生 Chrome 實例，負責執行、錄影、驗證這些前端任務。為了防止未授權操作和保護隱私，這個 subagent 會在完全隔離、類 sandbox 的 Chrome profile 裡跑。

這個 profile 不會共用或存取任何現有瀏覽器登入或私人 session（就像全新 incognito 環境），可以避免 context 洩漏或誤觸真實操作，同時讓 agent 安全地測試 UX 變更。

Debug prompt 不只限於靜態程式分析，內建瀏覽器還能逐步執行指令，直接指出畫面哪裡出錯、要怎麼修正。

3. 文件撰寫（The Author）在 SDD（規格驅動開發）裡，文件就是唯一的標準。如果文件和程式碼不同步，AI 很容易產生幻覺。要在 agent 的「skills」裡明確規定，README.md 或 CHANGELOG.md 裡的說明一定要保持更新。

Python 請用 Google Style Docstrings，TypeScript 請用 JSDoc。這些是有結構的註解寫法，能幫助 AI（和人類）不用逐行看程式邏輯，也能直接理解每個 function 的用途。

4. 資料工程（The Librarian）查詢資料表或搬移檔案時，就是資料工程師的角色。可以用像 Google Cloud Data Extension5 這類 IDE 擴充功能，直接在編輯器裡存取雲端資料。

要在 prompt 裡要求 agent 一定要顯示產生結果所用的 SQL 查詢或指令。

MCP：一次整合，支援所有框架

MCP 是由 Anthropic 開發，現在已經成為一個開放標準。大家常說它是「AI 工具界的 USB-C」，雖然有點誇張，但這個比喻蠻貼切的。你只要為你的資料庫（或 API、檔案系統）建一個 MCP server，任何支援 MCP 的 agent 都能直接使用，不用再寫專屬的整合程式。

建立一個 MCP Server

這裡有一個大約 40 行的範例 server，可以把 SQLite 資料庫公開出來：

# mcp_server.py — Exposes a SQLite database via MCP
import sqlite3
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent
server = Server("knowledge-base")
db = sqlite3.connect("knowledge.db")
@server.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="query_knowledge",
            description="Query the knowledge base with SQL",
            inputSchema={
                "type": "object",
                "properties": {
                    "sql": {
                        "type": "string",
                        "description": "SQL query to execute (SELECT only)"
                    }
                },
                "required": ["sql"]
            },
        ),
        Tool(
            name="add_knowledge",
            description="Add a new knowledge entry",
            inputSchema={
                "type": "object",
                "properties": {
                    "title": {"type": "string"},
                    "content": {"type": "string"},
                    "tags": {"type": "string", "description":
"Comma-separated tags"}
                },
                "required": ["title", "content"]
            },
        ),
    ]
@server.call_tool()
async def call_tool(name: tr, arguments: dict) -> list[TextContent]:
    if name == "query_knowledge":
        sql = arguments["sql"]
        if not sql.strip().upper().startswith("SELECT"):
            return [TextContent(type="text", text="Error: Only SELECT
queries allowed")]
        cursor = db.execute(sql)
        rows = cursor.fetchall()
        columns = [desc[0] for desc in cursor.description]
        result = [dict(zip(columns, row)) for row in rows]
        return [TextContent(type="text", text=str(result))]
    elif name == "add_knowledge":
        db.execute(
            "INSERT INTO knowledge (title, content, tags) VALUES (?, ?, ?)",
            (arguments["title"], arguments["content"], arguments.get("tags", ""))
        )
        db.commit()
        return [TextContent(type="text", text="Knowledge entry added.")]
async def main():
    async with stdio_server() as (read, write):
        init_options = server.create_initialization_options()
        await server.run(read, write, init_options)
if __name__ == "__main__":
    import asyncio
    asyncio.run(main())

團隊文化與流程演進

和現代 coding agent 合作，團隊的思維與文化也必須跟著調整。如果還想用舊有的流程配合新工具，就像把噴射引擎裝在馬車上，根本不可能飛起來。

這種技術沒辦法直接套用在二十年前的工作流上，然後期待它會有神奇的效果。想像一下，當你送出一個 pull request (PR) 結果變得超級龐大，開始考慮要不要拆成多個小 PR。

這時候，merge conflict 開始爆增，因為多位工程師同時動到同一份檔案，導致依賴鏈變得很難解開。PR #1 不能合併，因為還要等 PR #2，但 PR #2 又卡在 PR #3，而 PR #3 則被不同時區的 reviewer 擋住。

有些變更已經審核通過，相關的卻還在等，結果衍生出更多 conflict。最後，你突然發現自己卡在一個壞掉的 branch。這些問題可以分幾個類型來看：

  • Merge conflict：多位開發者在短時間內同時動到同一份檔案。
  • 審查塞車：一個超大 PR 又被拆成一堆巢狀 sub-PR，像俄羅斯娃娃一樣層層包裹。
  • 脈絡破碎：開發者離開時，隊友在共用檔案裡改了變數名稱；agent 引用的是舊快照，結果產生的程式碼還呼叫已經不存在的函式。

下面我們來看看，從這些狀況中學到的幾個經驗教訓。

程式碼審查

為了應付 AI 產生的 commit 量與規模，團隊必須思考全新的審查準則，才能避免過勞。雖然理想的工作流還有討論空間，以下這些做法可以幫助大家從「寫程式」順利轉型到「整合程式」：

高速整合的策略

  • 摘要與風險評估打包：建議每個 PR 都要附上一份 AI 產生的變更摘要、潛在出錯點，以及風險評估。這可以是一個 markdown 檔案，或是詳細的 commit 說明，讓人工審查者能把重點放在架構層級的影響，而不會被細節淹沒。
  • 重新定義擁有權：人工審查不再是挑剔 agent 產生、可隨時丟棄的程式風格，而要聚焦在架構藍圖的完整性。程式風格問題可以交給自動化整合工具處理，例如共用的語法檢查器或專案專屬的風格指南（像是 SKILLS.md）。
  • 「條件式 LGTM」：為了避免跨時區團隊因審查延遲 12 小時，可以考慮實施「條件式 LGTM」（Looks Good To Me）。審查者先有條件地通過 PR，只要所有自動化測試都過關，程式就自動合併。
  • 建立無責怪文化：在高速度環境下，產出最多程式碼的人常常成為 bug 或合併衝突的代罪羔羊。建議大家多討論，這些問題應該歸因於整合流程的設計，而不是單一使用 agent 的開發者。
  • 善用 Agent 程式碼審查：你可以打造一個 skill，讓它自動幫你做 code review，甚至還能寫 skill 來回應 code review。可以參考 Code Snippet 5.1。這類自動化可以透過 Github Actions6 或在 Github 上用 Gemini Code Assist7 來實現。

最後，你甚至可以思考討論：如果你能跟一群 agents 協作，團隊合作還有必要嗎？但如果你認為還是需要團隊，那可以嘗試把工作切分得更細，讓每個成員很少會動到同一個檔案，例如明確分配 API 與 UX 的負責人。

若真的有重疊，指定的「部分擁有者」就負責最後的同步整合。

部署會監控你 Repo 的 Agents

你可以寫出很棒的 review prompt，但誰會在每個 PR、每次夜間掃描時自動執行？難道每次都要人工按下按鈕嗎？

程式碼審查 skill 可以在 IDE 裡手動觸發。下一步，就是把它們部署成持續運作的程式碼分析 agent，也就是一種服務，會主動監控你的 repository，偵測事件（像是 PR 開啟、夜間排程），自動回報發現的問題，完全不需要人為介入。

這樣才能抓住週五下午疲憊的人類容易漏掉的東西：像是剛爆 CVE 的相依套件、六個月前的 TODO 變成安全漏洞。當你的團隊開始大量出貨 AI 產生的 PR 時，只有持續性的 reviewer 才能跟得上產出速度。

問題是，你需要多客製化？答案是一個光譜，分成三個層級，每個層級在「掌控度」與「簡便性」間做取捨。

這個光譜

這三個層級的差別在於誰擁有 runtime，誰負責撰寫審查標準：

  • 第一層 ， 受管型（例如 Gemini Code Assist on GitHub 8，或任何 SaaS PR reviewer）。這是現成的選擇。只要在 GitHub 或 GitLab 組織啟用，每個 PR 都會自動有 AI reviewer，針對風格、bug、安全性直接給建議。完全不用寫 prompt，也不用維護基礎設施，只要按人頭付費。缺點是，你只能用廠商的審查標準，不是你自己的。如果你的團隊有嚴格的內部風格或特殊領域風險（像 HIPAA、PCI、內部 SDKs），這種通用 reviewer 會漏掉關鍵細節。
  • 第二層 ， 混合型（例如 GitHub Action6 觸發 coding agent CLI ， Antigravity CLI12）。這是大多數團隊適合的中間路線。你可以自己寫 review skill（前面提到的 code-check.md 就是範例），把它 commit 到 repo，然後用 CI action 觸發你選定的 CLI，在非互動模式下執行，並把結果當作 PR comment 回報。runtime 屬於你的 CI 供應商，prompt、模型選擇、sandboxing、審查標準都由你掌控。
  • 第三層 ， 客製型（例如 Gemini Enterprise Agent Engine9 上的 ADK agent）。這是完全自有的路線。舉例來說，你可以把 reviewer 設定成 ADK agent，部署到 Agent Runtime，享有持久的 Session 與 Memory Bank，並串接原始碼主機的 webhook 事件。這種做法適合 reviewer 需要在多個 PR 重構期間持續保有 context，或要把「審查這個服務有沒有合規偏移」這類大任務，拆解成上百個子任務（用 planner 與 sub-agents pattern），甚至要透過 A2A10 跟其他 agent 協作。缺點是，你要自己負責 evaluation、observability、成本控管，以及 production 出問題時的 on-call。這種架構的實際案例可以參考 Siemens 案例研究17。

三個問題可以幫助你判斷該選哪一層：

1. 你的審查標準有多專屬？

  • 如果很通用 → 適合第一層。
  • 如果是團隊或 repo 專屬 → 適合第二或第三層。

2. agent 需不需要跨多次執行記住事情？

  • 不需要 → 適合第一或第二層。
  • 需要（像是 codebase 記憶、跨 PR context）→ 適合第三層。

3. 如果出錯，最嚴重的後果是什麼？

  • 只是多了一個吵雜的 comment → 任何層級都可以。
  • 如果會合併回歸 bug 或洩漏機密 → 第三層，並在每次 tool call 前加上前一節介紹的 Policy Server pattern。

多數團隊會在現成 reviewer 漏掉關鍵時，立刻知道自己該升級到哪一層。例如某家中型金融科技公司的平台團隊，一開始用第一層，結果發現合規 reviewer 老是標記 auditors 已經核准的樣板程式碼，卻漏掉唯一重要的問題：log 訊息裡未遮罩的 PII。

後來他們只花 40 行寫了個 compliance-check.md skill，接上 GitHub Action（第二層），一週內就大幅減少誤報。到目前為止，還沒遇到需要第三層的情境。

第三層全規模應用：原生圖形化程式碼理解

客製化程式碼審查 Runtime（第三層）不只是讓一個 ADK agent 看 diff 而已。這套架構還能支撐更重型的元件，適合真正需要審查上億行舊程式碼的 reviewer：像是用圖形資料庫儲存程式結構、用向量庫做語意檢索、用 sub-agent pipeline 拆解任務。

如果只是把程式碼當平面文字丟進 context window，這種規模很快就會爆掉；而標準的 RAG 又會把讓程式碼可讀的結構資訊全部剝除（像是哪個 class 屬於哪個檔案、function call 追溯到十年前的需求文件）。如果只用向量庫，這些結構關聯就消失了。

圖 1：客製化程式碼審查 Runtime 架構範例

在最大型舊系統現代化專案中，最常見的 pattern 是把 agent 建在知識圖譜之上。把程式碼、文件、ticket、設計 PDF 都匯入圖形資料庫（例如 Spanner Graph11），然後讓 agent 結合三種檢索模式：用圖形查詢（GQL）跑結構問題（像「所有間接呼叫 payment.process() 的 function」）、用向量搜尋（節點 embedding 的 ANN）找語意相關（像「找出跟這段描述相符的程式碼」）、用全文搜尋精確比對 identifier。

這樣結合起來，就能精準回答「我改這裡會影響什麼？」而不是只給一個自信但模糊的猜測。

第二個重點是任務拆解。只叫一個 agent「重構這個模組」，它一定會失敗。但如果用 ADK12 sub-agent pipeline 拆分：Search agent 先在圖形裡探索、Story agent 負責抓需求、Impact agent 預測副作用、Task-breakdown agent 產生最小單位的任務，最後才交給 Coding agent，這樣工作就變得有條有理。

這種方式已經在百萬行程式碼的生產環境裡試過，把同樣的重構工作從兩週縮短到幾小時。這就是第三層的全規模應用：不只是 agent 監控 PR，而是真正理解 PR 背後整個系統的 agent。

受管型程式碼審查 Runtime（第一層）讓你幾分鐘內就有通用 reviewer。混合型 Runtime 一天內就能有專屬 reviewer。客製型 Runtime（第三層）則能讓 reviewer 理解整個系統，但你要自主管理 runtime 和 evaluation。選擇能抓到你在意問題的最低層級就好。

永續性

現在我們遇到一個新現象：審核疲勞。根據 CNBC 報導引用 Quantum Workplace 的研究，經常使用 AI 的人，比沒用的人更容易有高燒盡感，機率高出 45% 14。

當開發者面對一連串細碎的審核請求（像是改一行程式、調整一次 tool call），很容易就變成機械式點「Approve」。這種低度疲勞會讓團隊為了跟上機器的速度，乾脆不再細看 AI 產出的內容，久了細節就容易被忽略。

為了保護團隊，可以從無止盡的監督，轉向有結構的界線管理：

  • 數位安靜時段：設定明確界線，讓審核請求不要延伸到晚上或週末。
  • Agent 洞察分享會：每週舉辦一次，讓開發者分享 AI 夥伴發現的模式，將零散的發現轉化成組織內的共同知識。

Zero-Trust Development：打造安全防護網

前面我們主要聚焦在程式碼審查和團隊文化。不過，還有一個很重要的教訓，就是當 agent 缺乏足夠 guardrails 時，會發生什麼事。

在一次例行的程式碼更新中，團隊發現了 Antigravity 內建 UI 瀏覽器的威力與限制。這個功能讓 AI agent 可以直接操作開發中的應用程式，不需要登入帳號密碼，對於 UX 測試來說非常方便。

但如果開啟 YOLO（自動通過）模式，agent 的動作甚至比人腦反應還快。

一個簡單的 prompt，要求建立一個按鈕，卻引發了一連串意外。browser agent 自動點擊了新按鈕，而這個按鈕原本是給 email agent 用的。

因為沒有指定 URL，agent 產生了幻覺，連上了一個早就淘汰、沒有 email 保護機制的舊 agent。結果呢？五十位同事都收到了充滿幻覺內容的假郵件。

這次事件凸顯了 context hallucination risk（脈絡幻覺風險）：當 AI 缺乏足夠資料時，會用 context 裡現有的字串來補洞，甚至包含硬編碼的 email 或 URL 這種敏感資訊。

如果只是寄出一封假郵件，或許還算小事。但想想 agent 的行為：它只是盡力用手邊的資料完成任務，卻沒檢查這樣做合不合理。這就是自主系統的核心風險。

如果沒有 human-in-the-loop 或 policy engine，agent 只會盡可能用現有資源達到目標。guardrails 不是選配，而是讓工具可控、避免失控的關鍵。

實作 Guardrails

當我們不斷突破 Agentic AI 的邊界時，會遇到一個矛盾：agent 必須有足夠的自主能力來解決複雜問題，但企業環境裡，卻無法承受它們「失控」的風險。

想像一下，有個 agent 被指派「解決客訴爭議」。為了有效執行任務，它需要存取客戶資料、email 工具和內部系統。但挑戰在於，如何確保它不會不小心寄信給整個資料庫，或意外洩漏專有程式碼。

自主 agent 的核心是 LLM，而 LLM 屬於機率型（非確定性）系統。把限制條件硬塞進 system prompt 很脆弱，context 經常爆滿，agent 也可能被 prompt injection 誘導繞過規則。

要打造可上線的生產級平台，必須有外部、不可竄改的治理機制。更多關於如何防護 agent 免於惡意程式攻擊，以及如何評估 agent 安全性的內容，可以參考第四天的論文：Vibe Coding Agent Security and Evaluation。

Sandboxing（沙箱隔離）

除了過濾字串之外，真正的安全性還需要一個受限的執行環境，也就是所謂的「Sandbox（沙箱）」，來限制 agent 的行為。即使你對輸出做了嚴格的過濾，LLM 還是有可能產生語法正確但邏輯上惡意的程式碼，進而危害主機系統。

透過在短暫、低權限的容器中執行 agent 任務，並且讓這些容器與主網路和敏感檔案系統隔離，就能設定一個「爆炸半徑」，有效保護核心基礎設施。這種做法下，即使 agent 被誘導執行破壞性指令，損害也只會侷限在可隨時清除重設的一次性實例，對主系統完全沒有影響。

在 Antigravity 裡，只要切換一個設定，就能啟用這層安全防護。進到 User Settings，開啟「Terminal Sandboxing」15功能即可。

如果你需要團隊用的可攜式雲端 sandbox，也可以把 agent 的工作區容器化。只要自訂一個 Dockerfile（例如放在 .gemini/sandbox.Dockerfile），以官方 Gemini CLI sandbox16 映像檔為基礎，再注入有限的雲端憑證，並在終端機設定 export GEMINI_SANDBOX=docker，就能讓 CLI 工具完全在 Docker 裡執行。

這兩種模式下，如果 agent 被誘導執行惡意指令，系統會直接在核心層遇到權限錯誤，主機系統完全不會受影響。

Human-in-the-Loop

雖然自動化是最終目標，但在高風險操作時，還是需要導入 Human-in-the-Loop（人類參與）機制，作為最後一道安全防線。這通常會針對特定風險等級的行動（像是部署程式到正式環境、修改資料庫 schema、或啟動金流交易）設置「檢查點」關卡。

系統會將 agent 經過過濾的意圖，呈現給人工審核人員，由他們手動確認。這樣一來，AI 的高效率可以和開發者的細膩判斷互補。最終，agent 可以負責繁重的工作，但架構完整性的最後把關，還是掌握在人類手中。

AI 產生的測試覆蓋率

隨著 AI 產生程式碼的浪潮來襲，測試工作也從人工手動，轉向 AI 自動產生測試覆蓋率，確保系統有可靠的安全網。在高開發速度的環境下，「測試驅動開發」不再只是理想，而是讓機器自己撰寫驗證自身產出的測試，真正落實自動化。

這個流程會要求 agent 先產生一個會失敗的單元測試，或是重現問題的指令（像是 curl 請求），然後才開始修正 bug。把這些自動化測試直接納入程式碼庫，每次快速迭代時，都有一套可驗證的檢查機制撐腰。

這樣一來，不僅能防止 bug 回流，也讓人工審查者更能信任自動化的「綠燈」結果，放心進行整合。

評估

為什麼用到 ML 模型的程式碼，需要特別的品質檢查？傳統的軟體測試，對於「產生」結果（generated）而不是「計算」結果（computed）的系統，其實不太夠用。

像 agent 或任何 ML 驅動的元件（例如分類器、摘要器、檢索器），就算它在所有工具的單元測試都過關，還是可能出大包：選錯工具、把關鍵答案改寫、甚至產生幻覺內容。

這種誤差範圍不是可以徹底消滅的缺陷，而是模型本身的特性，所以測試策略也必須能包容這種不確定性。

評估就是用分數判斷和容忍區間來取代傳統的二元斷言，補上這個缺口。單元測試只會問「這個函式有沒有回傳正確的值？」， 答案只有對或錯。但評估會問「agent 的表現，至少有沒有達到基準？

」這可以是 LLM 當評審，給 0 到 5 分的分數，也可以是檢查 agent 操作工具的順序，只要順序有合理變化就接受，或是設定一個品質門檻，只有分數低於這個門檻才會擋下來，不是只要斷言失敗就算錯。

傳統測試抓的是確定性的回歸，評估則能發現 行為漂移。

Policy Server

這裡有一個 Hybrid Policy Server（混合型政策伺服器）的範例，會在動作進入外部系統之前先進行攔截。它分成兩個層次運作：

  • 結構性閘門（紅綠燈機制）：這一層是根據角色和環境設定的確定性規則，屬於快速且二元的檢查（例如 viewer 角色不能用 send_email 工具）。這層可以直接擋下架構違規，不需要請 LLM 幫忙判斷。
  • 語意性閘門（智慧裁判）：這一層會用第二個、專門用途的 LLM（像是 Gemini）來檢查動作的意圖和內容，對照自然語言寫的隱私政策。這是針對「工具本身可以用，但用法違反政策」的情境。例如 admin 可以用 send_email，但不應該寄出未遮蔽的 PII（像是明文 email 或 API 金鑰）。這種狀況結構性規則就擋不住，因為你不可能用 regex 把所有 PII 洩漏都抓完。

這些規則會寫在標準的 policies.yaml 檔案裡：

environments:
  localhost:
    blocked_tools:
      - send_email
roles:
  viewer:
    allowed_tools:
      - list_files
      - read_file

範例片段 4：policies.yaml：這份設定檔會根據角色和環境，建立工具層級的確定性閘門規則。

下面是一個輕量級的 Policy Server 實作範例，專門攔截工具呼叫並在 runtime 驗證權限。

import os, yaml
from google.genai import Client
class PolicyService:
    def is_tool_allowed(self, tool_name: str) -> bool:
        # Check Environment Blocks
        env_config = self.config.get("environments", {}).get(self.env, {})
        if tool_name in env_config.get("blocked_tools", []): return False
        # Check Role Permissions
        role_allows = self.config.get("roles", {}).get(self.role, {}).
get("allowed_tools", [])
        return "*" in role_allows or tool_name in role_allows
    async def check_action_semantic(self, action_description: str) -> bool:
        client = Client(vertexai=True,
project=self.project_id, location=self.location)
        prompt = f"Evaluate if this action violates PII
policies: {action_description}"
        response =
client.models.generate_content(model="gemini-3.1-pro", contents=prompt)
        return not response.text.strip().upper().startswith("VIOLATION")

範例片段 5：policy_server.py：這個服務實作了一個混合型政策引擎，會攔截動作並進行結構性與語意性評估。

這段程式的運作流程如下：當 agent 決定要用某個工具時，執行流程會被攔截：

  1. 結構性檢查：這個角色／環境能不能用這個工具？（查 YAML）
  2. 語意性檢查：傳入的參數安全嗎？（問 Gemini，參考程式碼裡的 prompt），像未遮蔽的 email 就會被判定違規。
  3. 執行：兩關都過了才執行工具，否則就回傳「Policy Violation」訊息給 agent，讓它自己修正或優雅失敗。

這樣就建立了一層安全網，把執行邏輯和治理邏輯分開。這種關注點分離對企業軟體來說非常關鍵。

Context Hygiene 與 Prompt Sanitization

在自動化開發裡，一個很大的風險就是「Context Hallucination」（上下文幻覺）。當 agent 缺資料時，可能會隨便拿目前 context 裡的字串來補，結果就可能不小心洩漏像硬編碼 email 或私有 URL 這種敏感資訊。

為了防止這種狀況，要在 middleware 層落實嚴格的 Context Hygiene，像是 PII 遮蔽和 placeholder 注入。把個資都換成通用的 placeholder，確保 agent 運作時拿到的都是消毒過的資料。

此外，所有 agent 的輸出也都要經過 sanitization，避免 prompt injection 或 UI 被惡意操作，確保 AI 的「vibe」不會變成系統架構的漏洞。

實作動態 ContextResolver

這個安全設計可以用一個輕量級、基於 regex 的轉換工具來實現，直接寫在應用程式的核心程式碼裡。下面就是一個實際的實作範例：

這支 script 是核心的轉換引擎，會把像 [[VARIABLE_NAME]] 這種雙中括號格式的 placeholder 字串，動態換成 runtime 的參數或環境設定值：

import os
import re
from typing import Optional, Dict, Any
def resolve_context(template_str: str, override_state: Optional[Dict[str, Any]] =
None) -> str:
    """Scans a template string for [[VARIABLE_NAME]] and replaces it with
    values from override_state or os.environ.
    """
if template_str is None:
        return ""
    state_to_check = override_state or {}
    def replacement(match):
        var_name = match.group(1).strip()
        # 1. Prioritize runtime state overrides
        if var_name in state_to_check and state_to_check[var_name] is not None:
            return str(state_to_check[var_name])
        # 2. Fallback to validated environment variables
        elif var_name in os.environ and os.environ[var_name] is not None:
            return os.environ[var_name]
        # 3. Leave unresolved to prevent silent failures
        else:
            return match.group(0)
    # Resolve all bracketed variables dynamically, e.g., [[COMMENTER_EMAIL]]
    return re.sub(r'\[\[([^\]]+)\]\]', replacement, template_str)

範例片段 6：context_resolver.py：這個工具會把中括號包住的 placeholder 換成環境變數，維持 context hygiene。

為了全域落實 context hygiene，這個 sanitization 工具必須直接串進 agent 的執行 pipeline，當成驗證步驟。只要工具呼叫進來，就先攔截並動態消毒所有 prompt 注入的字串：

# ... inside the validate_tool_call framework execution ...
resolved_args = {}
for k, v in args.items():
    if isinstance(v, str):
        resolved_args[k] = resolve_context(v)
    elif isinstance(v, list):
        resolved_args[k] = [resolve_context(i) if isinstance(i, str) else i for i
in v]
    else:
        resolved_args[k] = v
args.clear()
args.update(resolved_args)

範例片段 7：tool_policy_engine.py：這個 middleware 把 context resolver 整合進 agent pipeline，讓工具參數在執行前都先消毒。

這樣一來，不管 agent 要執行什麼動作（像寄信或查詢雲端簡報），都會被攔截。引擎會把像 [[COMMENTER_EMAIL]] 或 [[DEFAULT_PRESENTATION_ID]] 這種 placeholder 自動換成授權的測試資產，安全且無聲地處理掉，完全不用在測試套件或 system prompt 裡硬編碼敏感 PII。

摘要

在不到一年的時間裡，軟體開發週期變得超級快。現在，agent 到中午就能產出上千行有完整註解的程式碼，這真的是一大突破。

不過，這種速度也帶來了一個重要的改變。AI 已經把「寫程式」這個瓶頸移除，現在的限制變成在下游：人類必須審查、測試、整合這些產出。這是大家一起分擔的認知負擔：人類像建築師一樣，撰寫 Test Specs、Integration Specs 和 MLOps/DevOps 藍圖，AI 則負責實際寫測試程式、做整合、處理細節的營運工作。

光靠更好的 prompt 或更快的模型，沒辦法解決整合的瓶頸。要成功，必須讓團隊合作模式持續進化，跟 agent 更精緻地協作，並且替這些永不休息的工具設下明確界線。

現在的挑戰，已經從單純產生程式碼，轉變為如何協調驗證、整合和部署這些工作的系統。

怎麼開始

如果你的團隊已經在 agent codebase 上實作這些模式，只要用 uv google-agents-cli setup 就能把七大技能安裝進你的 coding agent，包含 scaffolding、ADK 程式碼、生測、部署、發佈和 observability（可觀測性）。

這篇白皮書講到的模式，現在都能直接用指令執行：像 agents-cli scaffold 可以用 SDD（規格驅動開發）自動生成專案，agents-cli eval run 能讓 AI 幫你做測試覆蓋率的檢查，agents-cli deploy 則能把專案 sandbox 部署到 Cloud Run 或 Vertex AI Agent Engine。
